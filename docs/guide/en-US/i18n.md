# Internationalization (i18n)

EgoAdmin uses a dual-layer internationalization architecture across backend and frontend, supporting Chinese (zh-CN) and English (en-US).

## Overview

EgoAdmin's internationalization is split into backend and frontend layers. The backend uses the `go-i18n` library with embedded TOML message files, ensuring all API error messages are returned in the user's preferred language. The frontend uses `vue-i18n` to manage UI labels, button text, route titles, and other user-facing copy.

The two layers are connected via the `Accept-Language` HTTP header: the frontend sends the header based on the current locale, the Gateway converts it into gRPC metadata for downstream services, and the backend selects the matching message template accordingly.

The default language is Chinese (zh-CN). All features must maintain both Chinese and English translation files.

::: tip Core principle
Never return hardcoded user-facing strings. All text visible to users must go through the i18n mechanism.
:::

::: warning Product name exception
Product names and brand identifiers should not be translated. For example, "EgoAdmin" remains unchanged across all languages.
:::

## Core Usage

### Backend: TOML Message Files

Backend messages are defined in two TOML files and embedded at compile time:


### Backend: Error Wrapper Functions

The platform provides three error wrappers for different error types:

```go
import platformi18n "egoadmin/internal/platform/i18n"

// Business failure (400)
return platformi18n.ErrorFailed(ctx, "RoleInUseCount", map[string]any{"Count": count})

// Not logged in (401)
return platformi18n.ErrorNotLogin(ctx, "LoginExpired", nil)

// Access denied (403)
return platformi18n.ErrorAccessDenied(ctx, "DataPermissionOutOfScope", nil)
```

Plain text message retrieval (without returning an error):

```go
text := platformi18n.Message(ctx, "SomeMessageID")
text := platformi18n.Localize(ctx, "RoleInUseCount", map[string]any{"Count": count})
```

### Frontend: vue-i18n Usage

The frontend retrieves translated text via the `t()` function, supporting nested keys and interpolation:

```vue
<template>
  <el-button>{{ t('role.addSuccess') }}</el-button>
  <span>{{ t('common.total', { count: 10 }) }}</span>
</template>

<script setup lang="ts">
import { useI18n } from 'vue-i18n'

const { t } = useI18n()
</script>
```

Route and menu titles use i18n keys:

```ts
{
  path: '/role',
  meta: { title: 'menu.roleManagement' },
}
```

### Frontend: Language Switcher Component

The `LanguageSwitcher` component allows users to switch languages at runtime. Changes take effect immediately without a page refresh. The Element Plus component library's global locale is also updated, ensuring built-in components like date pickers and table pagination stay in sync.

```vue
<!-- Simplified example -->
<template>
  <el-dropdown @command="setLocale">
    <span>{{ currentLabel }}</span>
    <template #dropdown>
      <el-dropdown-menu>
        <el-dropdown-item
          v-for="opt in localeOptions"
          :key="opt.value"
          :command="opt.value"
        >
          {{ opt.label }}
        </el-dropdown-item>
      </el-dropdown-menu>
    </template>
  </el-dropdown>
</template>
```

**Step 2: Use error wrappers in the service layer**

```go
func (s *RoleService) DeleteRole(ctx context.Context, id uint64) error {
    role, err := s.repo.FindByID(ctx, id)
    if err != nil {
        return platformi18n.ErrorFailed(ctx, "RoleNotFound", nil)
    }

    count, err := s.repo.CountUsersByRoleID(ctx, id)
    if err != nil {
        return err
    }
    if count > 0 {
        return platformi18n.ErrorFailed(ctx, "RoleInUseCount",
            map[string]any{"Count": count})
    }

    if err := s.repo.Delete(ctx, id); err != nil {
        return err
    }
    return nil
}
```

## How It Works

### Language propagation chain

```
Frontend (vue-i18n)
  Ōåō Sets Accept-Language header
HTTP request ŌåÆ Gateway
  Ōåō Converts to grpcgateway-accept-language metadata
gRPC request ŌåÆ User service
  Ōåō Reads accept-language from metadata
Backend (go-i18n) ŌåÆ Selects matching localizer ŌåÆ Returns localized message
```

The frontend calls `getAcceptLanguage()` to get the Accept-Language value for the current locale, which is sent with every HTTP request. The Gateway converts it into gRPC metadata (`accept-language` or `grpcgateway-accept-language`), and downstream services use `localizer(ctx)` to read the language preference from the context.

### Backend localizer resolution

The `platformi18n.localizer(ctx)` function extracts Accept-Language from the context and creates an `i18n.Localizer` for that language. If the value is missing or empty, it falls back to `zh-CN`.

```text
// Simplified logic
func localizer(ctx context.Context) *i18n.Localizer {
    lang := extractAcceptLanguage(ctx) // reads from gRPC metadata
    if lang == "" {
        lang = defaultLanguage
    }
    return i18n.NewLocalizer(bundle, lang)
}
```

Message lookup flow:

1. Read the `accept-language` key from gRPC metadata in the context
2. If empty, try the `grpcgateway-accept-language` key
3. If still empty, use the default language `zh-CN`
4. Create an `i18n.Localizer` and call `Localize()` to get the message
5. If the message ID is not found, return the raw message ID as fallback

### Frontend language persistence

The frontend persists the language preference in `localStorage` (key: `egoadmin-locale`). On first load, the locale is determined by this priority:

1. Previously saved language in `localStorage`
2. Browser's `navigator.language`
3. Fallback to `zh-CN`

When switching languages, `setLocale()` updates the vue-i18n locale, writes to `localStorage`, and dispatches an `egoadmin-locale-changed` custom event to notify other components.

### Gateway error preservation

When the Gateway forwards errors from downstream services, it deserializes gRPC status details back into `*eerrors.EgoError`, preserving the original error message intact. This ensures that localized messages set by `platformi18n.ErrorFailed()` in downstream services are delivered to the frontend as-is.

## Common Issues

### Backend returns message ID instead of translated text

**Cause:** The message ID is missing from the TOML files, or the ID is misspelled.

**Solution:** Verify that both `active.zh-CN.toml` and `active.en.toml` contain the message ID. The `localizer` returns the raw message ID when no matching entry is found.

```go
// Fallback logic in message.go
msg, err := localizer(ctx).Localize(&i18n.LocalizeConfig{
    MessageID:    messageID,
    TemplateData: data,
})
if err != nil {
    return messageID // returns raw ID as fallback
}
```

### API errors still show the old language after switching locale

**Cause:** The Gateway is not forwarding `Accept-Language` to downstream gRPC services.

**Solution:** Verify that the Gateway's HTTP-to-gRPC middleware preserves `accept-language` metadata. Check that the frontend request includes the correct Accept-Language header.

### Parts of the page remain untranslated

**Cause:** Hardcoded strings are used instead of `t()`, or the language package is missing the corresponding key.

**Solution:** Search the codebase for hardcoded Chinese/English strings, replace them with `t('key')` calls, and ensure both language packages contain the key. Debugging steps:

```bash
# Search for potentially missed hardcoded Chinese text
grep -rn '[õĖĆ-ķ┐┐]' web/src/ --include='*.vue' --include='*.ts' | grep -v 'node_modules' | grep -v 'package/'
```

### Interpolation variables not rendering

**Cause:** Backend TOML uses <code v-pre>{{.VarName}}</code> (Go template syntax) while frontend uses `{varName}` (vue-i18n syntax). The two formats are different.

**Solution:** Backend templates use <code v-pre>{{.Count}}</code>, frontend language packages use `{count}`. Do not mix the two formats.

### Build error after adding new TOML messages

**Cause:** TOML files are embedded via `//go:embed` and included at compile time.

**Solution:** Simply rebuild with `go build` or `make build`. Embedded files are automatically included during compilation. No additional code generation step is needed.
