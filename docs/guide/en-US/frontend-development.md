# Frontend Development

The frontend app lives in `web/` and uses Vue 3, TypeScript, Element Plus, Pinia, Vue Router and Vite.

## Structure

```text
web/src/
├── api/                    # API instances, domain modules, api-manifest
│   ├── api-manifest.ts     # Generated by protoc-gen-api-catalog (read-only)
│   ├── index.ts            # Axios instance and interceptors
│   └── modules/            # API modules per backend service
├── assets/                 # Static assets, sprite sheets
├── buildfile/              # Plop code generation templates
├── components/             # Shared business components
├── config/                 # routeMenu configuration
├── i18n/                   # Internationalization
│   ├── index.ts            # vue-i18n initialization
│   └── package/            # Language packs (zh-CN.ts, en-US.ts)
├── layouts/                # Layouts
├── menu/                   # Menu configuration
├── router/                 # Router and guards
│   └── modules/            # Router modules per business area
├── store/                  # Pinia stores
│   └── modules/            # user, route, menu, etc.
├── styles/                 # Global styles, themes, SCSS resources
├── types/                  # Global type declarations
├── utils/                  # Utility functions
│   ├── login-crypto.ts     # Login encryption
│   └── heartbeat.ts        # Heartbeat keep-alive
└── views/                  # Page components
```

::: tip
`api-manifest.ts` is a generated artifact -- do not edit manually. After backend proto changes, run `make gen` and `pnpm run build` to update it.
:::

## TypeScript Configuration

The project uses strict mode TypeScript. Key settings:

```jsonc
{
  "compilerOptions": {
    "target": "ESNext",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "paths": {
      "@/*": ["./src/*"],
      "#/*": ["./src/types/*"]
    },
    "types": [
      "element-plus/global",
      "vite-plugin-pages/client",
      "vite-plugin-vue-meta-layouts/client"
    ]
  }
}
```

- `strict: true` enables all strict checks including `noImplicitAny` and `strictNullChecks`.
- `@/*` maps to `src/`, `#/*` maps to `src/types/`.
- The `types` array injects Element Plus global types, file-based routing types, and layout types.

::: warning
Do not use `any` types in code. If absolutely necessary, use `// eslint-disable-next-line` with a reason.
:::

### Dev Server Proxy

The dev server forwards requests via the `/proxy` prefix to the backend:

```ts
// vite.config.ts
server: {
  proxy: {
    '/proxy': {
      target: env.VITE_APP_API_BASEURL,
      changeOrigin: true,
      rewrite: (path: string) => path.replace(/\/proxy/, ''),
    },
  },
},
```

In production, Nginx reverse proxy handles routing -- the Vite proxy is not used.

### Global SCSS Resources

SCSS files under `src/styles/resources/` are automatically injected into every component's `<style>`. Sprite sheet SCSS files are also auto-loaded. No manual `@import` needed.

## API Calls

Business APIs use gRPC HTTP compatibility paths:

```ts
api.post('/user.v1.UserService/GetUserList', {
  page: 1,
  limit: 20,
})
```

Rules:

- Use `api.post`.
- Do not use RESTful GET/PUT/DELETE for business APIs.
- Use protobuf JSON field names.
- Treat backend `uint64` IDs as strings.

### Axios Instance (`api/index.ts`)

Core interceptor logic:

```ts
const instance = axios.create({
  baseURL: import.meta.env.VITE_APP_API_BASEURL,
  timeout: 30000,
})

// Request interceptor: inject token and accept-language
instance.interceptors.request.use((config) => {
  config.headers.Authorization = `Bearer ${getToken()}`
  config.headers['accept-language'] = getAcceptLanguage()
  return config
})

// Response interceptor: unified error handling
instance.interceptors.response.use(
  (res) => res.data,
  (err) => {
    // UNAUTHENTICATED / LOGIN_EXPIRED -> auto refresh token
    // Other errors -> ElMessage toast
  },
)
```

Error handling chain:

1. Detect gRPC status codes (`UNAUTHENTICATED`, `LOGIN_EXPIRED`).
2. Refreshable errors trigger automatic token refresh, then retry the original request.
3. Non-refreshable errors show an `ElMessage` error toast.
4. Login page and heartbeat errors are handled silently to avoid duplicate toasts.

### api-manifest.ts

Generated by `protoc-gen-api-catalog` from the OpenAPI spec, exports an `APIs` constant tree:

```ts
export const APIs = {
  user: {
    v1: {
      RoleService: {
        AddRole: 'user.v1.RoleService/AddRole',
        GetRoleList: 'user.v1.RoleService/GetRoleList',
      },
    },
  },
}
```

Used in routeMenu configuration to bind permissions. Build time automatically extracts them into `permission-contract.json`.

### API Module Convention

```ts
// web/src/api/modules/role.ts
import api from '../index'

export interface GetRoleListRequest {
  page: number
  limit: number
  name?: string
  status?: number
}

export function getRoleList(data: GetRoleListRequest) {
  return api.post('/user.v1.RoleService/GetRoleList', data)
}
```

- Each file corresponds to one backend service.
- Request/Response interfaces and call functions live in the same file.
- Paths use gRPC-compatible format: `/package.Service/Method`.

## Component Development Pattern

All page components use `<script setup lang="ts">`:

```vue
<script setup lang="ts">
import { computed, onMounted, reactive, ref } from 'vue'

// Reactive state
const loading = ref(false)
const formRef = ref<FormInstance>()

// Computed
const canEdit = computed(() => permissions.value.includes('EDIT'))

// Lifecycle
onMounted(() => {
  loadData()
})
</script>
```

Rules:

- Prefer `ref` and `reactive` over Options API.
- Use `computed` rather than `watch` for derived state.
- Name form refs as `formRef` with `FormInstance` type.
- Business components like `PageMain` and `SearchBar` are auto-registered by `unplugin-vue-components` -- no manual import needed.

## Internationalization

### Initialization

```ts
// i18n/index.ts
import { createI18n } from 'vue-i18n'
import zhCN from './package/zh-CN'
import enUS from './package/en-US'

const i18n = createI18n({
```

## Login Flow

Login uses RSA-OAEP encryption:

```text
1. User enters username and password
2. Frontend calls GetLoginCrypto to get public key and challenge
3. Frontend encrypts (username + password + challenge + nonce + timestamp) with the public key
4. Frontend sends passwordCipher + keyId + challengeId to the Login endpoint
5. Backend decrypts, verifies the challenge, and issues a JWT
```

Core code:

```ts
// utils/login-crypto.ts
export async function encryptPasswordPayload(
  payload: EncryptPasswordPayload,
): Promise<EncryptedPasswordPayload> {
  // 1. Get encryption parameters
  const challenge = await apiUser.getLoginCrypto({
    username: payload.username,
    ua: payload.ua,
    action: payload.action,
  })

  // 2. Import RSA public key
  const key = await importPublicKey(challenge.publicKey)

  // 3. Encrypt payload
  const plain = JSON.stringify({
    username: payload.username,
    password: payload.password,
    challengeId: challenge.challengeId,
    nonce: challenge.nonce,
    timestamp: Date.now(),
    ua: payload.ua,
    action: payload.action,
  })
  const cipher = await crypto.subtle.encrypt(
    { name: 'RSA-OAEP' },
    key,
    new TextEncoder().encode(plain),
  )

  // 4. Return ciphertext
  return {
    passwordCipher: arrayBufferToBase64(cipher),
    keyId: challenge.keyId,
    challengeId: challenge.challengeId,
  }
}
```

::: warning
Passwords must never be transmitted in plaintext. The frontend uses Web Crypto API with `RSA-OAEP` + `SHA-256`, algorithm fixed to `RSA-OAEP-SHA256`.
:::

## New Page Workflow

1. Add backend proto and implementation.
2. Add frontend API module.
3. Add page component under `views`.
4. Add router module.
5. Add routeMenu permission nodes.
6. Add i18n labels.
7. Run type check and build.

## List Page Pattern

A standard list page uses `PageMain` + `SearchBar` + `ElTable`:

```vue
<template>
  <PageMain>
    <!-- Search bar -->
    <SearchBar>
      <el-input v-model="listParam.name" clearable placeholder="Role Name" />
      <el-select v-model="listParam.status" clearable placeholder="Status">
        <el-option label="Enabled" :value="1" />
        <el-option label="Disabled" :value="2" />
      </el-select>
      <el-button type="primary" @click="loadData">Search</el-button>
      <el-button v-if="canAdd" type="primary" @click="handleAdd">Add</el-button>
    </SearchBar>

    <!-- Data table -->
    <el-table v-loading="loading" :data="tableData" border stripe>
      <el-table-column prop="name" label="Role Name" min-width="120" />
      <el-table-column prop="code" label="Role Code" min-width="120" />
      <el-table-column prop="status" label="Status" width="80">
        <template #default="{ row }">
          <el-tag :type="row.status === 1 ? 'success' : 'danger'">
            {{ row.status === 1 ? 'Enabled' : 'Disabled' }}
          </el-tag>
        </template>
      </el-table-column>
      <el-table-column label="Actions" width="200" fixed="right">
        <template #default="{ row }">
          <el-button v-if="canEdit" link type="primary" @click="handleEdit(row)">Edit</el-button>
          <el-button v-if="canDelete" link type="danger" @click="handleDelete(row)">Delete</el-button>
        </template>
      </el-table-column>
    </el-table>

    <!-- Pagination -->
    <el-pagination
      v-model:current-page="listParam.page"
      v-model:page-size="listParam.limit"
      :total="total"
      layout="total, sizes, prev, pager, next, jumper"
      @change="loadData"
    />
  </PageMain>
</template>
```

List page conventions:

- `listParam` uses `reactive` with `page`, `limit`, and search fields.
- `loadData` is called on `onMounted` and on the search button click.
- Action buttons use `userStore.VA(['ID'])` to control visibility.
- Use `el-pagination` for pagination.

## Form Page Pattern

Forms use `ElForm` with validation rules, typically inside a drawer or dialog:

```vue
<script setup lang="ts">
import type { FormInstance, FormRules } from 'element-plus'

const formRef = ref<FormInstance>()
const formData = reactive({
  name: '',
  code: '',
  status: 1,
  remark: '',
})

// Validation rules must match proto validate tags
const rules: FormRules = {
  name: [{ required: true, message: 'Role name is required', trigger: 'blur' }],
  code: [
    { required: true, message: 'Role code is required', trigger: 'blur' },
    { pattern: /^[A-Z_]+$/, message: 'Only uppercase letters and underscores', trigger: 'blur' },
  ],
  status: [{ required: true, message: 'Status is required', trigger: 'change' }],
}

async function handleSubmit() {
  await formRef.value?.validate()
  await addRole(formData)
  ElMessage.success('Operation successful')
}
</script>

<template>
  <el-form ref="formRef" :model="formData" :rules="rules" label-width="100px">
    <el-form-item label="Role Name" prop="name">
      <el-input v-model="formData.name" placeholder="Enter role name" />
    </el-form-item>
    <el-form-item label="Role Code" prop="code">
      <el-input v-model="formData.code" placeholder="Enter role code" />
    </el-form-item>
    <el-form-item label="Status" prop="status">
      <el-radio-group v-model="formData.status">
        <el-radio :value="1">Enabled</el-radio>
        <el-radio :value="2">Disabled</el-radio>
      </el-radio-group>
    </el-form-item>
    <el-form-item label="Remark" prop="remark">
      <el-input v-model="formData.remark" type="textarea" :rows="3" />
    </el-form-item>
  </el-form>
</template>
```

Form rules key points:

- `required` and `trigger` must match proto `validate` tags.
- The `prop` attribute must match the `formData` field name.
- Call `formRef.value?.validate()` before submitting.
- For edit scenarios, populate `formData` with `Object.assign`.

## ID Handling

```ts
const deptId = ref<string>('0')
const roleIds = ref<string[]>([])
```

Avoid converting backend IDs to JavaScript numbers. Protobuf `uint64` / `fixed64` fields are encoded as strings in JSON.

Pre-commit check:

```bash
rg -n "Number\\(|parseInt\\(|number\\[\\]|ref\\(0\\)" web/src
```
