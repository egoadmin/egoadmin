# API Design Standards

This page defines the gRPC API design standards for the EgoAdmin project, covering the full pipeline from Proto contracts to code generation, error handling, and OpenAPI documentation.

## Overview

EgoAdmin follows a **Proto-First** development model. All business APIs must start from Proto definitions. Developers generate Go stub code via `buf`, then implement the Controller layer logic. The HTTP compatibility layer is generated by the protoc-gen-go-http plugin, sharing the same Controller as gRPC. It serves as the access method for the frontend and external systems -- not as an independent RESTful API design.

gRPC is EgoAdmin's primary protocol. Inter-service communication (gateway <-> user, gateway <-> idgen) all uses gRPC. The externally exposed HTTP interfaces are implemented through protoc-gen-go-http generated HTTP-compatible handlers. All HTTP requests use the `POST` method with `body: "*"` mapping, and URL paths follow the `/<package.version.Service>/<Method>` format.

This page uses the **Role** module of the user service as its main thread, referencing real files such as `api/proto/user/v1/role.proto` and `api/proto-internal/idgen/v1/idgen.proto` to illustrate each standard's correct usage and anti-patterns.

## Core Usage

### 1. Proto-First Development Principle

**All business APIs must start from Proto definitions. Never add ad hoc business routes directly on Gin/EGO server.**

```go
// Anti-pattern: adding Gin routes directly for business APIs
// Never do this
func registerRoutes(r *gin.RouterGroup) {
    r.POST("/api/roles", func(c *gin.Context) { /* ... */ })
}

// Correct approach: business APIs are defined in proto files,
// HTTP routes are automatically generated by protoc-gen-go-http
```

EgoAdmin's HTTP access layer (protoc-gen-go-http handler in the gateway service) automatically maps `POST /api/<package.version.Service>/<Method>` to the corresponding gRPC method. Developers **do not need to manually register HTTP routes** -- they only need to define RPC methods in proto files with HTTP annotations.

### 2. Proto File Structure

Proto file locations:

| Type | Path | Purpose |
|------|------|---------|
| Public APIs | `api/proto/<service>/v1/<resource>.proto` | Externally exposed business interfaces |
| Internal APIs | `api/proto-internal/<service>/v1/<resource>.proto` | Inter-service internal calls |

::: tip
Public APIs are proxied through the gateway, generate OpenAPI documentation, and participate in permission management. Internal APIs are only for direct gRPC calls between services and do not generate HTTP mappings or OpenAPI docs.
:::

Each proto file follows a fixed structure pattern:

```protobuf
// api/proto/user/v1/role.proto
syntax = "proto3";

package user.v1;

import "google/api/annotations.proto";
import "google/api/field_behavior.proto";
import "google/protobuf/timestamp.proto";
import "protoc-gen-openapiv2/options/annotations.proto";
import "tagger/tagger.proto";

// Service-level OpenAPI documentation declaration
option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_swagger) = {
  info: {
    title: "RoleService"                         // Service name in Chinese
    description: "Service: user.v1.RoleService"  // Documentation reference
    version: "1.0"
    license: { name: "MIT" }
  }
  schemes: HTTP
  consumes: "application/json"
  produces: "application/json"
  // Bearer auth definition, shared by all protected interfaces
  security_definitions: {
    security: {
      key: "BearerAuth"
      value: {
        type: TYPE_API_KEY
        in: IN_HEADER
        name: "Authorization"
        description: "JWT auth header. In Swagger UI enter the full value: Bearer <token>"
      }
    }
  }
};

// Service definition
service RoleService {
  // Service tag (for OpenAPI grouping)
  option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_tag) = {
    description: "RoleService, service: user.v1.RoleService"
  };

  // Each RPC method requires HTTP mapping and OpenAPI annotations
  rpc AddRole(AddRoleRequest) returns (AddRoleResponse) {
    option (google.api.http) = {
      post: "/user.v1.RoleService/AddRole"
      body: "*"
    };
    option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
      tags: "RoleService, service: user.v1.RoleService"
      tags: "RoleService"
      description: "Auth: Requires Authorization: Bearer <token> and permission USER.V1.ROLESERVICE/ADDROLE."
      security: {
        security_requirement: { key: "BearerAuth" value: {} }
      }
    };
  }
}
```

Required imports for each proto file:

| Import | Purpose |
|--------|---------|
| `google/api/annotations.proto` | HTTP mapping annotations |
| `google/api/field_behavior.proto` | `REQUIRED` / `OPTIONAL` field behavior |
| `google/protobuf/timestamp.proto` | Timestamp type |
| `protoc-gen-openapiv2/options/annotations.proto` | OpenAPI documentation annotations |
| `tagger/tagger.proto` | Validation tags and copier tags |

### 3. Request/Response Message Design

**Each RPC method uses its own dedicated Request and Response messages -- never reuse them.**

```protobuf
// Correct: dedicated request for AddRole
message AddRoleRequest {
  Role role = 1 [
    (tagger.tags) = "validate:\"required\" label:\"Role Info\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = {
      title: "Role Info"
    },
    (google.api.field_behavior) = REQUIRED
  ];
}

message AddRoleResponse {
  uint64 id = 1;
}

// Wrong: multiple RPCs sharing one CreateRequest
// message CreateRequest { ... }  // AddRole and AddUser should not share this
```

**Field annotation trio**: validation tag + OpenAPI field title + field_behavior:

```protobuf
message GetRoleListRequest {
  // Validation tag: required + range constraint
  // OpenAPI title: used for doc generation
  // field_behavior: REQUIRED marker
  int32 page = 1 [
    (tagger.tags) = "validate:\"required,gte=1\" label:\"Page Number\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = {
      title: "Page Number"
    },
    (google.api.field_behavior) = REQUIRED
  ];

  int32 limit = 2 [
    (tagger.tags) = "validate:\"required,gte=1,lte=50\" label:\"Items Per Page\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = {
      title: "Items Per Page"
    },
    (google.api.field_behavior) = REQUIRED
  ];

  // Optional fields: validate tag starts with omitempty
  string sort = 3 [
    (tagger.tags) = "validate:\"omitempty,gte=1\" label:\"Sort Field\""
  ];

  string order = 4 [
    (tagger.tags) = "validate:\"omitempty,oneof=desc asc\" label:\"Sort Order\""
  ];
}
```

**Time fields always use `google.protobuf.Timestamp` with copier tags for Go struct field name mapping:**

```protobuf
// Role shared entity message (referenced by multiple Request/Response messages)
message Role {
  uint64 id = 1 [(tagger.tags) = "copier:\"ID\""];
  google.protobuf.Timestamp created_at = 2 [(tagger.tags) = "copier:\"CreatedAtToRPC\""];
  google.protobuf.Timestamp updated_at = 3 [(tagger.tags) = "copier:\"UpdatedAtToRPC\""];
  google.protobuf.Timestamp deleted_at = 4 [(tagger.tags) = "copier:\"DeletedAtToRPC\""];
  string name = 5 [
    (tagger.tags) = "validate:\"required\" label:\"Role Name\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Role Name" },
    (google.api.field_behavior) = REQUIRED
  ];
  int32 data_perm = 7 [
    (tagger.tags) = "validate:\"required,oneof=1 2 3 4\" label:\"Data Permission\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Data Permission" },
    (google.api.field_behavior) = REQUIRED
  ];
}
```

::: warning
Request/Response messages must be defined independently and never reused across RPCs. Shared entities (such as `Role`, `User`) can be referenced by multiple RPC Request/Response messages, but Request/Response messages themselves cannot be shared.
:::

### 4. HTTP Mapping Rules

All EgoAdmin RPCs uniformly use the `POST` method with `body: "*"` mapping. The URL pattern is:

```
POST /<package.version>.<ServiceName>/<MethodName>
```

Examples:

```protobuf
// User login
rpc Login(LoginRequest) returns (LoginResponse) {
  option (google.api.http) = {
    post: "/user.v1.UserService/Login"
    body: "*"
  };
}

// Role list
rpc GetRoleList(GetRoleListRequest) returns (GetRoleListResponse) {
  option (google.api.http) = {
    post: "/user.v1.RoleService/GetRoleList"
    body: "*"
  };
}
```

::: danger
RESTful URL path parameter designs are prohibited. Do not write patterns like `get: "/roles/{id}"` or `put: "/roles/{id}"`. EgoAdmin's HTTP compatibility layer uniformly uses the POST + body approach, which aligns with gRPC semantics and simplifies frontend calls.
:::

### 5. Error Handling Standards

Each service defines error enums in `api/proto/<service>/v1/errors.proto`, using the `@plugins=protoc-gen-go-errors` plugin to auto-generate Go error codes:

```protobuf
// api/proto/user/v1/errors.proto
syntax = "proto3";
package user.v1;

// @plugins=protoc-gen-go-errors
// ErrorUser errors
enum ErrorUser {
  // @code=INTERNAL internal service error
  ERROR_USER_UNKNOWN_UNSPECIFIED = 0;
  // @code=UNKNOWN duplicate user import
  ERROR_USER_IMPORT = 1;
  // @code=UNKNOWN organization cannot be deleted
  ERROR_USER_DEPT_NOT_DEL = 2;
}
```

The Controller layer maps domain errors to i18n errors for return:

```go
// internal/app/user/controller/role_grpc.go
func mapRoleError(ctx context.Context, err error) error {
    var inUse roledomain.InUseError
    switch {
    case err == nil:
        return nil
    case errors.Is(err, roledomain.ErrNameExists):
        return platformi18n.ErrorFailed(ctx, "RoleNameExists", nil)
    case errors.As(err, &inUse):
        return platformi18n.ErrorFailed(ctx, "RoleInUseCount",
            map[string]any{"Count": inUse.Count})
    case errors.Is(err, roledomain.ErrInUse):
        return platformi18n.ErrorFailed(ctx, "RoleInUse", nil)
    default:
        return err
    }
}
```

Error handling pipeline: **Domain layer returns domain error -> Controller's mapXxxError function matches -> Converts to i18n error -> Returns to client as gRPC status**.

### 6. Validation and Copier Tags

**Validation tags (validate)** are embedded by the tagger plugin in generated code, and the Validator middleware automatically enforces them at the Controller entry point:

```protobuf
// Common validation rules
string name = 1 [(tagger.tags) = "validate:\"required,gte=2,lte=50\""];
string order = 2 [(tagger.tags) = "validate:\"omitempty,oneof=desc asc\""];
uint64 id = 3 [(tagger.tags) = "validate:\"required,gte=1\""];
string password = 4 [
  (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = {
    extensions: {
      key: "x-go-tag-validate"
      value: { string_value: "omitempty,gte=6,lte=32" }
    }
  }
];
```

**Copier tags** control the field name mapping from Proto to Go struct:

```protobuf
// Copier tags map to Go struct field names
uint64 id = 1 [(tagger.tags) = "copier:\"ID\""];             // -> Go: ID
google.protobuf.Timestamp created_at = 2
    [(tagger.tags) = "copier:\"CreatedAtToRPC\""];           // -> Go: CreatedAtToRPC
```

The Controller layer uses `copier.Copy` to convert store/domain models to Proto responses:

```go
func (s *RoleGRPC) GetRole(ctx context.Context, in *userv1.GetRoleRequest) (
    out *userv1.GetRoleResponse, err error) {
    out = &userv1.GetRoleResponse{Role: &userv1.Role{}}

    role, err := s.role.GetRole(ctx, in.GetId())
    if err != nil {
        return
    }

    // copier automatically maps field names based on proto copier tags
    if err = copier.Copy(&out.Role, &role); err != nil {
        return
    }
    return
}
```

### 7. Code Generation

Generate all code in one command:

```bash
make gen  # Equivalent to: make gen.proto gen.proto.internal gen.go gen.wire
```

Step-by-step generation:

```bash
# Public proto: buf lint + buf generate, outputs to api/gen/go and api/gen/openapi
make gen.proto

# Internal proto: buf lint + buf generate, outputs to api/gen/go
make gen.proto.internal

# go generate: annotation code, mocks, etc.
make gen.go

# Wire dependency injection: optional SERVICE=<service> for single service
make gen.wire
make gen.wire SERVICE=user
```

Generated artifacts:

| Directory | Content |
|-----------|---------|
| `api/gen/go/<service>/v1/` | Go protobuf generated code (gRPC server/client interfaces, message structs) |
| `api/gen/openapi/` | OpenAPI YAML files (embedded by gateway) |
| `internal/app/<service>/server/wire_gen.go` | Wire dependency injection generated code |

::: danger
Never hand-edit any generated files. All files under `api/gen/` and `wire_gen.go` will be overwritten on the next `make gen`. To make changes, modify the corresponding `.proto` files or Wire injector files.
:::

### 8. OpenAPI Documentation

The gateway embeds `openapi.yaml` at startup and serves a Swagger UI at runtime. Each RPC's OpenAPI annotations control the description, grouping, and security requirements in the documentation.

**Public interface (no authentication required):**

```protobuf
rpc Login(LoginRequest) returns (LoginResponse) {
  option (google.api.http) = {
    post: "/user.v1.UserService/Login"
    body: "*"
  };
  option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
    tags: "UserService, service: user.v1.UserService"
    tags: "UserService"
    description: "Auth: Public interface, no Authorization required."
    // No security block = no authentication needed
  };
};
```

**Protected interface (Bearer Token required):**

```protobuf
rpc GetMenus(GetMenusRequest) returns (GetMenusResponse) {
  option (google.api.http) = {
    post: "/user.v1.UserService/GetMenus"
    body: "*"
  };
  option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
    tags: "UserService, service: user.v1.UserService"
    tags: "UserService"
    description: "Auth: Requires Authorization: Bearer <token>, no specific permission needed."
    security: {
      security_requirement: { key: "BearerAuth" value: {} }
    }
  };
};
```

**Protected interface with permission control:**

```protobuf
option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
  description: "Auth: Requires Authorization: Bearer <token> and permission USER.V1.ROLESERVICE/ADDROLE."
  security: {
    security_requirement: { key: "BearerAuth" value: {} }
  }
};
```

::: warning
All protected interfaces must have both the `security` block and the permission description. Omitting the security declaration will cause Swagger UI to let users debug the interface without authentication, which is inconsistent with actual runtime behavior.
:::

## Real-World Examples

### Example: Adding a "Notice Management" API

The following demonstrates the complete flow from Proto to Controller for adding a Notice resource to the user service.

**Step 1: Define the Proto**

```protobuf
// api/proto/user/v1/notice.proto
syntax = "proto3";
package user.v1;

import "google/api/annotations.proto";
import "google/api/field_behavior.proto";
import "google/protobuf/timestamp.proto";
import "protoc-gen-openapiv2/options/annotations.proto";
import "tagger/tagger.proto";

option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_swagger) = {
  info: { title: "NoticeService" description: "Service: user.v1.NoticeService" version: "1.0" }
  schemes: HTTP
  consumes: "application/json"
  produces: "application/json"
  security_definitions: {
    security: {
      key: "BearerAuth"
      value: { type: TYPE_API_KEY in: IN_HEADER name: "Authorization"
               description: "JWT auth header. In Swagger UI enter the full value: Bearer <token>" }
    }
  }
};

service NoticeService {
  option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_tag) = {
    description: "NoticeService, service: user.v1.NoticeService"
  };

  // AddNotice creates a notice
  rpc AddNotice(AddNoticeRequest) returns (AddNoticeResponse) {
    option (google.api.http) = { post: "/user.v1.NoticeService/AddNotice" body: "*" };
    option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
      tags: "NoticeService, service: user.v1.NoticeService"
      tags: "NoticeService"
      description: "Auth: Requires Authorization: Bearer <token> and permission USER.V1.NOTICESERVICE/ADDNOTICE."
      security: { security_requirement: { key: "BearerAuth" value: {} } }
    };
  };

  // GetNoticeList retrieves the notice list
  rpc GetNoticeList(GetNoticeListRequest) returns (GetNoticeListResponse) {
    option (google.api.http) = { post: "/user.v1.NoticeService/GetNoticeList" body: "*" };
    option (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_operation) = {
      tags: "NoticeService, service: user.v1.NoticeService"
      tags: "NoticeService"
      description: "Auth: Requires Authorization: Bearer <token> and permission USER.V1.NOTICESERVICE/GETNOTICELIST."
      security: { security_requirement: { key: "BearerAuth" value: {} } }
    };
  };
}

message AddNoticeRequest {
  Notice notice = 1 [
    (tagger.tags) = "validate:\"required\" label:\"Notice Info\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Notice Info" },
    (google.api.field_behavior) = REQUIRED
  ];
}

message AddNoticeResponse {
  uint64 id = 1;
}

message GetNoticeListRequest {
  int32 page = 1 [
    (tagger.tags) = "validate:\"required,gte=1\" label:\"Page Number\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Page Number" },
    (google.api.field_behavior) = REQUIRED
  ];
  int32 limit = 2 [
    (tagger.tags) = "validate:\"required,gte=1,lte=50\" label:\"Items Per Page\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Items Per Page" },
    (google.api.field_behavior) = REQUIRED
  ];
}

message GetNoticeListResponse {
  int32 total = 1;
  repeated Notice notices = 2;
}

message Notice {
  uint64 id = 1 [(tagger.tags) = "copier:\"ID\""];
  google.protobuf.Timestamp created_at = 2 [(tagger.tags) = "copier:\"CreatedAtToRPC\""];
  string title = 3 [
    (tagger.tags) = "validate:\"required,gte=1,lte=100\" label:\"Title\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Title" },
    (google.api.field_behavior) = REQUIRED
  ];
  string content = 4 [
    (tagger.tags) = "validate:\"required\" label:\"Content\"",
    (grpc.gateway.protoc_gen_openapiv2.options.openapiv2_field) = { title: "Content" },
    (google.api.field_behavior) = REQUIRED
  ];
}
```

**Step 2: Generate Code**

```bash
make gen.proto       # Generate Go stubs and OpenAPI docs
make gen.go          # Generate go generate artifacts
make gen.wire        # Regenerate Wire injection
```

**Step 3: Implement the Controller**

After generation, the `NoticeService` gRPC server interface appears in `api/gen/go/user/v1/`. The Controller layer implements this interface:

```go
// internal/app/user/controller/notice_grpc.go
package controller

import (
    "context"

    userv1 "github.com/egoadmin/egoadmin/api/gen/go/user/v1"
    "github.com/egoadmin/egoadmin/internal/app/user/service"
    "github.com/egoadmin/egoadmin/internal/user/internal/store"
    "github.com/egoadmin/egoadmin/internal/user/internal/auditlog"
    "github.com/jinzhu/copier"
)

type NoticeGRPC struct {
    notice *service.NoticeService
    logger auditlog.Loger
}

func NewNoticeGRPCController(
    notice *service.NoticeService,
    logger auditlog.Loger,
) *NoticeGRPC {
    return &NoticeGRPC{notice: notice, logger: logger}
}

func (s *NoticeGRPC) AddNotice(ctx context.Context,
    in *userv1.AddNoticeRequest) (out *userv1.AddNoticeResponse, err error) {
    out = &userv1.AddNoticeResponse{}

    defer func() {
        if err == nil {
            s.logger.Save(ctx, "System Management - Notice", "Create", "Create notice", in)
        }
    }()

    notice := &store.NoticeModel{
        Title:   in.GetNotice().GetTitle(),
        Content: in.GetNotice().GetContent(),
    }

    if err = s.notice.AddNotice(ctx, notice); err != nil {
        return
    }

    out.Id = notice.ID
    return
}

func (s *NoticeGRPC) GetNoticeList(ctx context.Context,
    in *userv1.GetNoticeListRequest) (out *userv1.GetNoticeListResponse, err error) {
    out = &userv1.GetNoticeListResponse{}

    notices, total, err := s.notice.GetNoticeList(ctx, int(in.GetPage()), int(in.GetLimit()))
    if err != nil {
        return
    }

    out.Total = int32(total)
    out.Notices = make([]*userv1.Notice, 0, len(notices))
    if err = copier.Copy(&out.Notices, &notices); err != nil {
        return
    }
    return
}
```

**Step 4: Register with Wire and Server**

Add `NewNoticeGRPCController` to the `ProviderSet` in `controller.go`, register `NoticeService` at the server layer, and update the Wire injector file before running `make gen.wire`.

**Step 5: Configure Permissions and Update Frontend**

For gateway-visible APIs, register the permission entry `USER.V1.NOTICESERVICE/ADDNOTICE` in the permission system and connect the frontend routes and pages.

::: tip
For the complete end-to-end API addition flow, refer to the [API Development Workflow](/en-US/guide/api-development) page, which covers steps from Proto to frontend.
:::

### Internal API Example: idgen Service

Internal APIs do not require HTTP mappings or OpenAPI annotations, resulting in a simpler structure:

```protobuf
// api/proto-internal/idgen/v1/idgen.proto
syntax = "proto3";
package idgen.v1;

import "google/protobuf/timestamp.proto";

// Note: no openapiv2_swagger option, no HTTP mapping
service SegmentService {
  // EnsureSegment creates a segment definition
  rpc EnsureSegment(EnsureSegmentRequest) returns (EnsureSegmentResponse);
  // AllocateSegment allocates a non-overlapping half-open range [start, end)
  rpc AllocateSegment(AllocateSegmentRequest) returns (AllocateSegmentResponse);
  // Health checks whether the allocator is available
  rpc Health(SegmentServiceHealthRequest) returns (SegmentServiceHealthResponse);
}

message AllocateSegmentRequest {
  string namespace = 1;     // Namespace isolates ID streams
  string name = 2;          // Business ID stream name
  int64 requested_step = 3; // Desired segment size
}

message AllocateSegmentResponse {
  int64 start = 1;   // First allocated ID
  int64 end = 2;     // One past the last allocated ID
  int64 step = 3;
  int64 min_step = 4;
  int64 max_step = 5;
  int32 status = 6;
}
```

Key differences between internal and public APIs:

| Dimension | Public APIs (`api/proto/`) | Internal APIs (`api/proto-internal/`) |
|-----------|---------------------------|--------------------------------------|
| HTTP mapping | Yes, protoc-gen-go-http compatible | No, direct gRPC calls |
| OpenAPI docs | Generated | Not generated |
| Permission mgmt | Participates in Casbin | Does not participate |
| Generation cmd | `make gen.proto` | `make gen.proto.internal` |
| Buf config | `buf.gen.yaml` | `buf.gen.internal.yaml` |

## How It Works

### protoc-gen-go-http Mapping Mechanism

When a Proto defines HTTP annotations, the `protoc-gen-go-http` plugin generates HTTP-compatible registration code. The gateway service registers these handlers at startup, deserializing HTTP POST requests into Proto messages, directly calling the same Controller's gRPC method, then serializing the response as JSON to return. Since the HTTP handler and gRPC server share the same Controller, there is no reverse proxy and no extra latency.

Request pipeline:

```text
Browser / Frontend
  -> HTTP POST /api/user.v1.RoleService/AddRole
  -> Gateway protoc-gen-go-http handler
  -> Deserialize JSON -> AddRoleRequest Proto
  -> Directly call the same Controller's AddRole method (no proxy)
  -> Receive AddRoleResponse Proto
  -> Serialize to JSON and return
```

### Validation Middleware Pipeline

The `tagger.tags` validation tags on Proto fields are embedded as Go struct tags during code generation. The `*validate.Validate` middleware injected at the Controller layer automatically reads struct tags at method entry and performs validation. Validation failures immediately return a gRPC InvalidArgument error without reaching the Service/Domain layers.

## Common Issues

**Q: Can I use GET requests to retrieve data in EgoAdmin?**

No. EgoAdmin uniformly uses POST with `body: "*"` and does not use GET or path parameters. This aligns with gRPC semantics (all calls are method calls) and simplifies authentication and logging. All frontend requests send JSON bodies via POST.

**Q: Will adding new fields break old clients?**

Proto3's default value design ensures forward compatibility. When adding new fields, use new field numbers. Old clients ignore unknown fields, and new fields appear as zero values to old clients. **Never delete or rename existing field numbers**, as this will break deserialization compatibility.

**Q: Why does each RPC need its own dedicated Request/Response message?**

Independent messages ensure each method's interface contract is clear, preventing field bloat or implicit coupling between methods. When a method needs additional parameters, it does not affect other methods' signatures.

**Q: What is the relationship between the OpenAPI security block and the permission description in the description field?**

The `security` block controls whether Swagger UI displays the authentication input box. The `description` field's permission text is documentation for human readers. Both must be in sync: protected interfaces need both the `security` block (so Swagger UI correctly shows authentication) and the permission description in the description (so developers know what specific permissions are needed).

**Q: What are the naming conventions for Proto file names?**

- File names use lowercase snake_case: `role.proto`, `auth_internal.proto`
- Public API files go under `api/proto/<service>/v1/`
- One file per resource, named in singular form
- Internal API files go under `api/proto-internal/<service>/v1/`, optionally with an `_internal` suffix

**Q: How do I handle Proto field to Go struct field name mismatches?**

Use the `copier` tag. For example, the Proto field `created_at` (snake_case) is called `CreatedAtToRPC` (PascalCase) in the Go struct. Annotate it as `[(tagger.tags) = "copier:\"CreatedAtToRPC\""]` in the Proto. The Controller layer's `copier.Copy` will automatically complete the mapping based on the tag.

**Q: Can I define Streaming RPCs in Proto?**

EgoAdmin currently does not use Streaming RPCs. All business APIs are Unary RPCs (request-response pattern). If file uploads or real-time push are needed in the future, evaluate using the TUS protocol or an independent WebSocket channel rather than adding stream methods to existing Proto services.
