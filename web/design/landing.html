EgoAdmin 不是只带登录和 CRUD 的后台模板，而是一套面向 Go 微服务的开发底座。它用 Proto First 串起接口、验证、文档、权限和响应映射，同时沉淀统一文件能力、契约化权限、分布式 ID、认证会话、数据库迁移和 AI 友好的工程规则。

定义 proto，就完成半个接口工程

service、rpc、request、response 和字段标签会继续生成 HTTP/gRPC 双栈入口、参数验证、OpenAPI 文档、API manifest、权限标识和响应映射依据。

文件能力不是一个上传接口

统一处理 multipart、TUS 断点续传、秒传、对象存储、file_reference、Range 下载和图片动态处理，业务只需要绑定 referenceId。

权限简单，但边界完整

接口契约、routeMenu、permission-contract、Casbin 和角色授权形成闭环，开发前中期不用在业务代码里到处补权限胶水。

设计能力

EgoAdmin 提前沉淀的后台系统基础设施

把派生项目里反复重写、最容易出错的底层能力，固化为契约驱动、边界清晰的工程底座。

CONTRACT

Proto First 契约驱动

业务接口从 proto 开始，而不是从临时 HTTP handler 或页面按钮开始，减少多处重复定义造成的不一致。

  • 同一 RPC 同时生成 gRPC 服务和 POST HTTP 兼容路径
  • validate、label、field_behavior 和 OpenAPI 注解共同描述请求约束
  • copier 标签和转换方法明确 ORM/model 到响应结构的映射
FILE

统一上传、下载和图片处理

把后台项目最容易散落的文件生命周期收进 gateway 和 upload/cdn 组件。

  • multipart、TUS 断点续传、秒传预检和对象存储直写
  • /cdn/file/ 支持认证、签名、inline 和 Range 下载
  • file_object、file_reference、upload_session 管理临时、绑定、释放和清理
RBAC

契约化权限和数据范围

权限不靠散落的字符串和人工记忆维护，而是由接口、菜单、按钮和角色授权链路约束。

  • API catalog、routeMenu 和 permission-contract 约束角色可授权边界
  • DataScope 支持全部、本组织及以下、本组织、仅本人等数据范围
  • 普通管理员不能创建或授予超出自身范围的角色权限
IDGEN

稳定高效的分布式 ID

内置 idgen 服务，避免每个派生项目重复处理 Snowflake、号段、机器号和 ID 外露问题。

  • 号段模式提供高性能全局 ID 分配
  • 机器号租约和关闭释放适合多服务部署
  • idcodec 支持内部数字 ID 到外部字符串 ID 的可逆编码
BASE

微服务工程底座

默认 gateway、user、idgen 三个服务，保留真实微服务项目需要的启动、治理和迁移链路。

  • gateway 统一承接 HTTP 兼容、前端内嵌、上传下载和 API 聚合
  • Atlas + GORM model 按服务数据库边界生成版本化迁移
  • readiness、health check、graceful shutdown 和资源关闭顺序已内置
AI

AI 编程友好

项目把容易出错的开发顺序写成稳定规则，让 AI 更容易按正确层次扩展业务。

  • EgoAdmin skill 固化 proto、权限、迁移、上传和验证规则
  • 新增能力按 proto → 生成 → 实现 → 权限 → 验证 推进
  • make gen、前端 build、service check 和 e2e 让改动可验证
开发流水线

新增业务能力时的默认路径

每一步都落在正确的层，改动始终可生成、可授权、可验证。

定义 proto

接口、请求响应、校验标签、OpenAPI 说明和 copier 映射一起设计。

make gen

生成 gRPC/HTTP 兼容代码、OpenAPI、API catalog 和前端 API manifest。

实现三层

controller、service、store：业务规则、事务、数据权限和响应转换留在正确层。

绑定 routeMenu

页面、按钮和 API 权限进入 permission-contract，角色授权受契约约束。

运行验证

前端构建、服务测试、迁移校验和 gateway e2e 覆盖完整链路。
